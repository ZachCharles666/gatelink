## 2. 当前问题

### 2.1 注册流程问题

- 注册成功后，页面立刻展示完整 `buyer_api_key`
- 用户还没有建立“账号体系”和“工作台”的认知，就先被迫理解 API Key
- 注册页当前同时承担了：
  - 创建账号
  - 自动登录
  - 发放 API Key
  - 一次性展示密钥

这会让新用户第一印象更像“工具页”，不像平台型产品。

### 2.2 登录流程问题

- 登录成功后，当前页面只提示“登录成功，buyer_token 已写入浏览器”
- 没有自动进入真正的“已登录后台”
- 登录后的成就感和目标感不明显

### 2.3 买家后台问题

当前买家后台更像“技术状态页”，而不是面向购买与管理的 SaaS 控制台：

- 导航仍偏基础能力页：
  - 充值
  - 用量明细
  - API Key
- 缺少平台核心商业场景：
  - 商品广场
  - 搜索与筛选
  - 已购资产
  - 订单/期限/状态管理
  - 用户中心

## 3. 重构目标

本次重构建议把买家端体验改成两层结构：

1. `认证流程`
   - 注册只负责完成账号创建
   - 登录才进入工作台

2. `已登录后的 SaaS 后台`
   - 左侧导航
   - 顶部全局栏
   - 右上角个人中心
   - 围绕“浏览商品、购买、管理已购资源、管理 API Key、查看账单/用量”展开

## 4. 目标体验

## 4.1 注册成功后的目标流程

建议改为：

1. 用户提交注册信息
2. 页面提示“注册成功”
3. 主按钮引导去登录页
4. 不在注册成功页直接展示完整 API Key

推荐文案：

- 标题：`注册成功`
- 说明：`账号已创建完成，请前往登录以进入买家后台。`
- 主按钮：`去登录`
- 次按钮：`返回首页`

设计理由：

- 注册完成时，用户首先应该获得“账号建立成功”的反馈
- API Key 属于“已登录后的资源管理动作”，更适合放到工作台内

## 4.2 登录成功后的目标流程

建议改为：

1. 用户登录成功
2. 自动跳转到买家后台默认首页
3. 默认首页不再是“技术说明页”，而是“买家工作台”

建议默认落地页：

- 首选：`广场`
- 备选：`工作台总览`

我的推荐是：**默认进入广场**

原因：

- 这是买家最核心的动作入口
- 能最快建立“这是一个可以买资源的平台”的认知
- 比直接进入余额页或 cURL 示例页更符合商业路径

## 5. 买家后台信息架构建议

## 5.1 整体布局

建议采用标准 SaaS 控制台布局：

- 左侧：主导航
- 顶部：全局状态栏
- 右上角：用户个人中心
- 主区域：当前模块页面

### 顶部栏建议

右上角展示：

- 头像或默认用户图标
- 用户昵称 / 手机号 / 邮箱脱敏展示
- 余额快捷入口
- 个人中心菜单

个人中心菜单建议包括：

- 个人资料
- 安全设置
- API Key 管理
- 账单与充值
- 退出登录

## 5.2 左侧导航建议

建议第一版导航结构如下：

1. `广场`
2. `我的购买`
3. `API Key 管理`
4. `充值与账单`
5. `用量分析`
6. `个人中心`

## 6. 各页面职责建议

## 6.1 广场

这是买家登录后的第一核心页面。

### 页面目标

让买家可以浏览、搜索、筛选平台上的可购买资源。

### 页面建议内容

- 搜索框
- 筛选器
  - 厂商
  - 模型
  - 有效期
  - 价格区间
  - 状态
- 资源列表卡片 / 表格

### 建议展示字段

- 商品标题
- 模型编号
- 厂商
- 可用时效 / 截止时间
- 计费方式
- 售价
- 状态
- 资源说明
- 购买按钮

### 特别建议

从产品文案上，可以写成“API Key”“资源”“套餐”“授权计划”。

但从系统实现边界上，**不建议把卖家的真实上游 Key 原样暴露给买家**。  
更稳妥的做法是：

- 广场卖的是“可购买访问资源”
- 买家买到的是“平台内可用的访问资格 / 交付凭证”
- 平台仍然负责对真实上游 key 做托管和代理

也就是说：

- 前台可以叫“买 Key”
- 后台实现最好仍然是“买访问权”，而不是把 seller 原始 key 明文交付

## 6.2 我的购买

这是第二核心页面。

### 页面目标

让买家管理已经购买的资源和到期信息。

### 建议展示字段

- 购买时间
- 生效时间
- 到期时间
- 厂商
- 模型
- 订单号 / 唯一 ID
- 当前状态
- 剩余额度 / 使用情况
- 关联 API Key 或交付凭证

### 页面能力建议

- 搜索
- 状态筛选
- 复制标识
- 查看详情
- 查看使用记录
- 到期提醒

### 状态建议

- 待生效
- 生效中
- 即将到期
- 已过期
- 已停用

## 6.3 API Key 管理

这个页面应该从“注册成功页”中独立出来。

### 页面目标

让买家在已登录状态下管理平台发放给自己的 API Key。

### 建议能力

- 查看当前 API Key 状态
- 复制 API Key
- 重置 API Key
- 查看最近一次重置时间
- 查看调用示例
- 查看权限范围 / 绑定资源

### 设计建议

- 不要在首次注册成功时强行展示完整 Key
- 改成在这里统一管理
- 可以保留“完整 Key 只展示一次”的安全策略
- 若用户重置 Key，则旧 Key 立即失效

## 6.4 充值与账单

页面建议合并当前“充值”与后续“支付记录/账单”能力。

### 建议内容

- 当前余额
- 充值入口
- 充值记录
- 消费账单
- 支付状态

## 6.5 用量分析

保留现有用量页，但从“列表页”升级为“分析页”。

### 建议内容

- 总调用次数
- 总消费金额
- 厂商维度分布
- 模型维度分布
- 时间趋势
- 明细列表

## 6.6 个人中心

### 建议内容

- 基本资料
- 联系方式
- 修改密码
- 安全设置
- 通知设置
- 发票 / 账单偏好

## 7. 推荐的用户主路径

建议的买家主路径如下：

1. 访问注册页
2. 注册成功
3. 跳转登录页
4. 登录成功
5. 进入“广场”
6. 浏览资源
7. 购买
8. 在“我的购买”中查看已购资源
9. 在“API Key 管理”中复制或重置平台 API Key
10. 在“用量分析”和“账单”中持续管理使用情况

## 8. 与当前架构的关系

这次重构不是纯 UI 调整，它会影响产品语义。

### 需要保留的当前能力

- 买家认证
- 买家余额
- 充值申请
- 用量记录
- API Key 重置

### 需要新增的能力

- 广场列表
- 广场搜索/筛选
- 商品详情
- 购买流程
- 已购资源列表
- 订单与到期信息
- 个人中心信息

### 建议注意

当前仓库里的买家端 API 主要覆盖：

- 认证
- 余额
- 充值
- 用量
- API Key 重置

因此如果要实现你提到的“广场”和“我的购买”，后端还需要新增对应的数据模型和接口能力。

## 9. 推荐的第一阶段落地范围

为了降低改造成本，我建议分两阶段。

### Phase A：先优化注册/登录与后台骨架

优先做：

1. 注册成功后不再展示 API Key
2. 注册成功后引导去登录
3. 登录成功后直接跳转到买家后台
4. 买家后台改成标准 SaaS 布局
5. 左侧先落基础导航框架
6. 右上角补个人中心入口

这一阶段不用先把“购买系统”全做完，也能明显提升体验。

### Phase B：补齐平台商业路径

再继续做：

1. 广场
2. 搜索 / 筛选
3. 资源详情
4. 购买流程
5. 我的购买
6. 到期与状态管理

## 10. 对当前前端实现的直接建议

### 注册页

当前：

- 注册成功后显示完整 API Key
- 自动写入 token 和 api_key 到浏览器

建议：

- 注册成功后不直接展示 API Key
- 页面只展示“注册成功，请登录”
- 可考虑仅保留注册成功标志，不自动进入已登录态

### 登录页

当前：

- 登录成功后留在原页显示成功提示

建议：

- 登录成功后直接跳转买家后台
- 优先跳转到 `广场`

### 买家后台壳子

当前：

- 顶部横向导航
- 更像“能力页集合”

建议：

- 改成左侧导航 + 顶部用户区
- 提升为标准 SaaS 后台结构

## 11. 结论

这次优化的核心不是“把页面做得更漂亮”，而是把买家端从“技术能力展示”升级成“完整购买与管理后台”。

最重要的三个方向是：

1. **注册成功后，不再立刻展示 API Key，而是引导去登录**
2. **登录成功后，直接进入一个真正的已登录 SaaS 后台**
3. **买家后台以“广场 + 我的购买 + API Key 管理 + 个人中心”为核心组织**

如果按产品优先级来排，我建议下一步就按这个顺序推进：

1. 先改注册成功页
2. 再改登录成功跳转
3. 再重构买家后台壳子
4. 最后补广场和我的购买
