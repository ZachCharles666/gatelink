## 2. 当前问题

### 注册流程

- 注册成功后立即展示完整 `buyer_api_key`
- 用户在建立账号认知之前，就先被要求理解 API Key
- 注册页当前同时承担“注册、自动登录、发放 key、展示 key”四个职责

### 登录流程

- 登录成功后只停留在登录页展示提示
- 没有自动进入真正的已登录后台

### 买家后台

- 当前更像技术状态页
- 缺少平台型产品应有的购买与管理结构

## 3. v0.2 目标体验

### 注册成功后

- 注册只负责完成账号创建
- 注册成功页提示用户前往登录
- 不在注册成功页直接完整展示 API Key

推荐文案：

- 标题：`注册成功`
- 说明：`账号已创建完成，请前往登录以进入买家后台。`
- 主按钮：`去登录`

### 登录成功后

- 登录成功后直接进入买家后台
- 默认落地页建议为 `广场`

原因：

- 广场是买家最核心的业务入口
- 比落在余额页或技术说明页更符合平台认知

## 4. 买家后台信息架构

### 布局

- 左侧：导航栏
- 顶部：全局信息栏
- 右上角：个人中心
- 主区域：当前模块

### 左侧导航建议

1. `广场`
2. `我的购买`
3. `API Key 管理`
4. `充值与账单`
5. `用量分析`
6. `个人中心`

### 右上角个人中心建议

- 头像或默认图标
- 邮箱 / 手机号脱敏展示
- 余额快捷入口
- 下拉菜单：
  - 个人资料
  - 安全设置
  - API Key 管理
  - 账单与充值
  - 退出登录

## 5. 页面职责

### 5.1 广场

目标：

- 浏览平台可购买资源
- 搜索、筛选、比较并发起购买

建议展示字段：

- 商品标题
- 模型编号
- 厂商
- 可用时效 / 截止时间
- 计费方式
- 售价
- 状态
- 资源说明

建议能力：

- 搜索
- 按厂商/模型/有效期/价格筛选
- 查看详情
- 购买

### 5.2 我的购买

目标：

- 管理已购买的资源、订单、状态与时效

建议展示字段：

- 购买时间
- 生效时间
- 到期时间
- 厂商
- 模型
- 订单号 / 唯一 ID
- 当前状态
- 使用情况
- 关联凭证信息

### 5.3 API Key 管理

目标：

- 管理平台分配给买家的 API Key

建议能力：

- 查看当前 Key 状态
- 复制 Key
- 重置 Key
- 查看最近重置时间
- 查看调用示例

说明：

- 不建议在首次注册成功页直接强行展示完整 Key
- API Key 的完整展示与安全提醒应放到已登录后的管理页

### 5.4 充值与账单

目标：

- 管理余额、充值申请、充值记录与消费账单

### 5.5 用量分析

目标：

- 从“明细页”升级为“分析页”
- 支持总量、趋势、模型分布、厂商分布与明细

### 5.6 个人中心

目标：

- 管理资料、安全设置、通知偏好与账户偏好

## 6. 关键产品边界

从产品表达上，可以让买家理解为“购买 API Key / 购买资源”。

但从平台实现边界看，不建议：

- 直接把卖家的真实上游密钥明文交付给买家

更稳妥的实现应是：

- 平台托管上游资源
- 买家购买的是“访问资格 / 交付凭证 / 平台管理下的调用能力”
- 真正的上游 key 仍由平台管理

## 7. 推荐落地顺序

### Phase A

先做体验结构调整：

1. 注册成功页改成“去登录”
2. 登录成功后跳转买家后台
3. 买家后台改成左侧导航 + 顶部个人中心

### Phase B

再补业务主路径：

1. 广场
2. 我的购买
3. API Key 管理增强
4. 充值与账单
5. 用量分析增强
